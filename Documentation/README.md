# Safe Signal Performance Simulator

A browser-based load-testing and simulation tool for the Safe Signal platform. It creates synthetic cases and enrolled devices, then simulates GPS and telemetry updates at configurable intervals.

## What It Does

The simulator authenticates against a Safe Signal backend, creates cases with realistic-looking data, enrolls a device against each case, and then drives location / activation / battery updates via the mobile-device telemetry API.

## Data Sources: Server vs Local

### Data Fetched Live from the Server

When you authenticate, the simulator queries the backend for master data. If the server is unavailable or the API fails, it falls back to safe defaults.

| Field | Source | Endpoint | Fallback |
|---|---|---|---|
| **Victim gender** | Server master data | `GET /api/master/genders` | `Female`, `Male` |
| **Perpetrator gender** | Server master data | `GET /api/master/genders` | `Female`, `Male` |
| **Case language** | Server master data | `GET /api/languages?for_case=true` | `en-GB` |

**Important distinction:** The server maintains *two separate language lists*:

- `GET /api/languages?for_case=true` – Languages supported by the **victim smartphone app**. The simulator uses this because every simulated case represents a victim carrying a device.
- `GET /api/languages?for_interface=true` – Languages supported by the **case-management web UI** (staff/admin interface). The simulator does **not** use this.

### Data Generated Locally

All other case fields are generated by the simulator. None of them represent real people, vehicles, or addresses.

| Field | How It Is Generated | Notes & Biases |
|---|---|---|
| **First name** | Random pick from a hardcoded list (`~80 female`, `~80 male`) | The lists are heavily Irish/Celtic (Aisling, Niamh, Cian, Padraig) with a sprinkling of international names. They are **not** an even demographic cross-section. |
| **Last name** | Random pick from a hardcoded list (`~90 surnames`) | Predominantly Irish (Murphy, O'Brien, Walsh) and British surnames. Not globally representative. |
| **Officer name** | Random pick from a hardcoded list (`6 fictional names`) | e.g. "Sergeant Hynes", "DS Murray". Completely fictional. |
| **Town / city** | Random pick from a hardcoded list (`~15 towns`) | Mix of UK and Republic of Ireland towns (Belfast, Galway, Brighton, Slough). |
| **Street** | Random pick from a hardcoded list (`~7 fictional street names`) | Generic English-sounding street names (Harper Lane, Beacon Crescent). |
| **Postcode** | Synthetically generated from real UK components | Uses genuine UK postcode areas (AL, B, BA, SW, NW, etc.) and valid inward-letter combinations, but the parts are randomly assembled. **Not geocoded** to the chosen town – the postcode will not match the address in reality. |
| **Vehicle make & model** | Random pick from a hardcoded catalog (`8 makes × 4 models`) | Real UK-market cars (Ford Focus, BMW 320i, Toyota Yaris, etc.). |
| **Vehicle colour** | Random pick from a hardcoded list (`8 colours`) | Blue, Black, Silver, Red, White, Green, Grey, Navy. |
| **Vehicle registration (VRM)** | Synthetic UK-style plate | Two letters + two digits + space + three letters (e.g. `AB12 XYZ`). Not checked against DVLA. |
| **Case photos** | Randomly chosen from local `casephotos/` folders | Two folders: `male/` and `female/`. A photo is picked purely by gender folder; **no attempt is made to match ethnicity, age, or appearance to the generated name**. |
| **Risk level** | Random pick from `['High', 'Medium', 'Low']` | Uniform distribution. |
| **Risk category IDs** | Random subset of `[1, 2, 3, 4, 5]` | Each ID has a 40% chance of inclusion. These map to risk categories configured on the server. |
| **Disability IDs** | Random subset of `[1, 2, 3, 4]` | Each ID has a 40% chance of inclusion. Maps to server disabilities list. |
| **Warning marker IDs** | Random subset of `[1, 2, 3, 4, 5, 6]` | Each ID has a 40% chance of inclusion. Maps to server warning-markers list. |
| **Device OS type** | Random pick from `['Android', 'Apple']` | 50/50 split. |
| **Phone number** | Synthetic UK mobile number | Always starts with `07` followed by 8 random digits (e.g. `07612345678`). Not a real number. |
| **Device location** | Clustered around organisation geocenter, or random UK/IRL region | If the chosen organisation has a geo-centre, devices start nearby and drift. Otherwise they are placed around random lat/lng clusters (London, Manchester, Birmingham, Dublin, etc.). |
| **Review date** | Random future date (21–90 days ahead) | - |
| **Date of birth** | Random date ~25–55 years in the past | Used for perpetrator when a perpetrator is added. |

## Known Biases & Limitations

1. **Name ethnicity bias** – First and last names skew Irish/British/Celtic. If your service area is more diverse, the synthetic names will not reflect that.
2. **Town coverage** – Only ~15 towns are represented; most of the UK and Ireland is absent.
3. **Postcode mismatch** – Postcodes look real but are not matched to the street or town. A case may list "Belfast" with a "SW1" postcode.
4. **Photo mismatch** – Stock photos are assigned by gender folder only; a case named "Priya Nguyen" may receive a photo of a person who does not visually match that name.
5. **Vehicle realism** – The car catalog is small and UK-market centric.
6. **Perpetrator probability** – A perpetrator is added to ~35% of cases regardless of risk level or other factors.
7. **Gender randomness** – Both victim and perpetrator genders are now uniformly random from the server's gender list. Previously the simulator hard-coded a 90% female bias for victims.
8. **No real PII** – Every field is synthetic. If you see a number or name that matches reality it is purely coincidental.

## Quick Start

1. Open `simulator/index.html` (or `indexinone.html`) in a browser.
2. Enter your API endpoint (defaults to `/`).
3. Log in with a Safe Signal staff account.
4. Set device count and timing, then click **Start**.

The simulator will:
- Fetch genders and victim-app languages from the server.
- Create cases with synthetic data.
- Enroll a device per case.
- Begin sending location / battery / activation updates.

Use **Stop** to archive or delete the created cases (configurable via *Teardown Mode*).
